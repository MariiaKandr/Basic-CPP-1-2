## Исключения

### Проблема обработки ошибок

Представьте, что мы пишем функцию, которая должна прочитать данные из файла, обработать их и вернуть результат. В процессе работы может произойти множество ошибок:

- Файл не найден.
- Нет прав на чтение.
- Данные повреждены (неверный формат).
- Не хватает памяти для выделения буфера.

Как сообщить об ошибке вызывающему коду? Традиционные подходы:

#### 1. Коды возврата

```cpp
int readData(const char* filename, Data& outData) {
    FILE* f = fopen(filename, "r");
    if (!f) return -1;           // Ошибка открытия файла
    
    char buffer[256];
    if (fgets(buffer, sizeof(buffer), f) == NULL) {
        fclose(f);
        return -2;               // Ошибка чтения
    }
    
    // Парсим данные...
    if (parseError) {
        fclose(f);
        return -3;               // Ошибка формата
    }
    
    fclose(f);
    return 0;                    // Успех
}

int main() {
    Data d;
    int result = readData("input.txt", d);
    if (result == -1) {
        std::cerr << "Файл не найден\n";
    } else if (result == -2) {
        std::cerr << "Ошибка чтения\n";
    } else if (result == -3) {
        std::cerr << "Неверный формат\n";
    } else {
        // Работаем с d
    }
}
```

**Проблемы кодов возврата:**
- **Смешивание логики и обработки ошибок**: Код функции забит проверками и `return`.
- **Легко забыть проверить**: Компилятор не заставляет проверять код возврата.
- **Неудобно возвращать значения**: Функция должна либо возвращать результат, либо код ошибки. Приходится использовать выходные параметры (как `outData`), что неудобно.
- **Сложно пробрасывать ошибку вверх**: Если `readData` вызывается из `processFile`, а та из `main`, каждому уровню нужно передавать код ошибки дальше.

#### 2. Глобальная переменная `errno` (стиль C)

Многие функции C (например, `fopen`) устанавливают глобальную переменную `errno`. Это ещё хуже: её легко забыть проверить, она может быть перезаписана другим вызовом, и не потокобезопасна.

#### 3. Механизм исключений — решение

Исключения позволяют **отделить** код, который обнаруживает ошибку, от кода, который её обрабатывает. Когда возникает ошибка, функция **выбрасывает исключение** (`throw`). Выполнение текущей функции немедленно прекращается, и управление передаётся вверх по стеку вызовов, пока не найдётся подходящий **обработчик** (`catch`). При этом автоматически вызываются деструкторы всех локальных объектов, что гарантирует освобождение ресурсов (благодаря RAII).

### 1. Базовый синтаксис: `throw`, `try`, `catch`

#### Простейший пример

```cpp
#include <iostream>
#include <stdexcept>

double divide(double a, double b) {
    if (b == 0.0) {
        // Бросаем исключение. Здесь создаётся объект исключения.
        throw std::runtime_error("Деление на ноль!");
    }
    return a / b;
}

int main() {
    try {
        // Код, в котором может произойти исключение
        double result = divide(10.0, 0.0);
        std::cout << "Результат: " << result << std::endl;  // Не выполнится
    }
    catch (const std::runtime_error& e) {
        // Обработка конкретного типа исключения
        std::cerr << "Ошибка: " << e.what() << std::endl;
    }
    catch (const std::exception& e) {
        // Обработка любого стандартного исключения
        std::cerr << "Стандартное исключение: " << e.what() << std::endl;
    }
    catch (...) {
        // Обработка любого исключения (не рекомендуется, если не знаете, что делать)
        std::cerr << "Неизвестное исключение" << std::endl;
    }
    
    std::cout << "Программа продолжается после обработки" << std::endl;
    return 0;
}
```

**Разберём по шагам:**

1. **`throw`** создаёт объект исключения (в данном случае `std::runtime_error`) и передаёт управление обработчику.
2. **`try` блок** содержит код, в котором ожидается возможное исключение.
3. **`catch` блоки** перехватывают исключения определённых типов. Они проверяются по порядку, как `if-else if`. Пойманное исключение доступно через ссылку (всегда ловите по константной ссылке, чтобы избежать копирования и срезки).
4. Если исключение не поймано в `main`, вызывается `std::terminate()`, и программа аварийно завершается.

#### Как работает раскрутка стека (stack unwinding)

Рассмотрим более сложный пример с несколькими функциями:

```cpp
#include <iostream>
#include <string>

class Tracer {
public:
    Tracer(const std::string& name) : name_(name) {
        std::cout << "Конструктор " << name_ << std::endl;
    }
    ~Tracer() {
        std::cout << "Деструктор " << name_ << std::endl;
    }
private:
    std::string name_;
};

void level3() {
    Tracer t3("level3");
    std::cout << "level3: бросаем исключение!" << std::endl;
    throw std::runtime_error("Ошибка на глубине 3");
}

void level2() {
    Tracer t2("level2");
    level3();
    std::cout << "level2: это не выполнится" << std::endl;
}

void level1() {
    Tracer t1("level1");
    try {
        level2();
    } catch (const std::runtime_error& e) {
        std::cout << "level1: поймали исключение: " << e.what() << std::endl;
        // Можно перебросить исключение дальше
        // throw;
    }
}

int main() {
    try {
        level1();
    } catch (...) {
        std::cout << "main: поймали что-то" << std::endl;
    }
    return 0;
}
```

**Вывод:**
```
Конструктор level1
Конструктор level2
Конструктор level3
level3: бросаем исключение!
Деструктор level3
Деструктор level2
level1: поймали исключение: Ошибка на глубине 3
Деструктор level1
```

**Важнейший момент:** Когда исключение пролетает через `level3` и `level2`, **автоматически вызываются деструкторы** всех локальных объектов (`t3`, затем `t2`). Это и есть **раскрутка стека** (stack unwinding). Именно благодаря этому RAII и умные указатели так хорошо работают с исключениями: память и другие ресурсы освобождаются автоматически, без необходимости писать ручные проверки.

### 2. Стандартные классы исключений

В заголовочном файле `<stdexcept>` определена иерархия стандартных классов исключений. Все они наследуются от `std::exception`, который имеет виртуальный метод `what()`, возвращающий `const char*` с описанием ошибки.

Упрощённая иерархия:

```
std::exception
├── std::logic_error          (ошибки программиста, можно предотвратить проверкой)
│   ├── std::invalid_argument
│   ├── std::domain_error
│   ├── std::length_error
│   └── std::out_of_range
├── std::runtime_error        (ошибки времени выполнения, трудно предсказать)
│   ├── std::range_error
│   ├── std::overflow_error
│   └── std::underflow_error
└── ... (есть ещё std::bad_alloc, std::bad_cast и др.)
```

**Как использовать:**

```cpp
#include <stdexcept>
#include <vector>
#include <iostream>

int getElement(const std::vector<int>& v, size_t index) {
    if (index >= v.size()) {
        // Логическая ошибка: выход за границы
        throw std::out_of_range("Индекс " + std::to_string(index) + 
                                " вне диапазона [0, " + std::to_string(v.size()) + ")");
    }
    return v[index];
}

void setAge(int age) {
    if (age < 0 || age > 150) {
        throw std::invalid_argument("Возраст должен быть от 0 до 150");
    }
    // ...
}

int main() {
    std::vector<int> vec = {1, 2, 3};
    try {
        std::cout << getElement(vec, 5) << std::endl;
    } catch (const std::out_of_range& e) {
        std::cerr << "Ошибка доступа: " << e.what() << std::endl;
    }
}
```

Вы можете создавать свои классы исключений, наследуя от `std::exception` или его потомков:

```cpp
class FileOpenError : public std::runtime_error {
public:
    explicit FileOpenError(const std::string& filename)
        : std::runtime_error("Не удалось открыть файл: " + filename) {}
};

// Использование:
// throw FileOpenError("data.txt");
```

### 3. Исключения и умные указатели / RAII

Вспомним пример с сырыми указателями и проблемой утечки при исключении. Теперь посмотрим, как умные указатели решают её элегантно.

#### Опасный код с сырыми указателями

```cpp
void processFile(const char* filename) {
    FILE* f = fopen(filename, "r");
    if (!f) throw std::runtime_error("Файл не открыт");
    
    char* buffer = new char[1024];   // Выделили память
    
    if (fgets(buffer, 1024, f) == nullptr) {
        delete[] buffer;             // Обработка ошибки: надо освободить память
        fclose(f);                   // и закрыть файл
        throw std::runtime_error("Ошибка чтения");
    }
    
    // ... работа с данными ...
    
    delete[] buffer;
    fclose(f);
}
```

**Проблема:** При исключении нужно вручную освобождать все ресурсы. Легко что-то забыть, особенно если ресурсов много.

#### Безопасный код с RAII и умными указателями

```cpp
#include <memory>
#include <cstdio>
#include <stdexcept>

// Простая RAII-обёртка для FILE*
class FileHandle {
    FILE* f_;
public:
    explicit FileHandle(const char* filename) : f_(fopen(filename, "r")) {
        if (!f_) throw std::runtime_error("Не удалось открыть файл");
    }
    ~FileHandle() { if (f_) fclose(f_); }
    FILE* get() const { return f_; }
    // Запрет копирования (или можно реализовать перемещение)
    FileHandle(const FileHandle&) = delete;
    FileHandle& operator=(const FileHandle&) = delete;
};

void processFileSafe(const char* filename) {
    FileHandle file(filename);                           // RAII для файла
    std::unique_ptr<char[]> buffer(new char[1024]);     // Умный указатель для буфера
    
    if (fgets(buffer.get(), 1024, file.get()) == nullptr) {
        throw std::runtime_error("Ошибка чтения");
        // НИКАКИХ РУЧНЫХ ОСВОБОЖДЕНИЙ! Деструкторы FileHandle и unique_ptr
        // будут вызваны автоматически при раскрутке стека.
    }
    
    // ... работа с данными ...
}
```

Теперь код не только короче, но и **абсолютно безопасен**: при любом исключении файл будет закрыт, память освобождена. Это и есть главная сила RAII в сочетании с исключениями.

### 4. Стратегии обработки исключений

#### Где ловить?

- **На границах модуля** (например, в `main`, в потоке, в callback из C API). Это предотвращает "вылет" исключения туда, где его не ждут (например, в код на C).
- **На уровне, где можно осмысленно отреагировать**: повторить операцию, показать пользователю ошибку, записать в лог и продолжить, использовать резервный вариант.

#### Перебрасывание исключений

Иногда нужно частично обработать исключение (например, записать в лог) и передать его дальше:

```cpp
try {
    // ...
} catch (const std::exception& e) {
    logError(e.what());
    throw;   // Перебрасываем ТО ЖЕ САМОЕ исключение (сохраняя тип)
}
```

Обратите внимание на `throw;` без аргументов — он перебрасывает текущее исключение. Если написать `throw e;`, то произойдёт копирование и возможная срезка.

#### Гарантии безопасности исключений

При проектировании функций и классов важно думать, как они ведут себя при исключениях. Различают три уровня гарантий:

1. **No-throw guarantee (без исключений)**: Функция не бросает исключений никогда. Примеры: деструкторы, `swap`, простые геттеры. В C++11 появилось ключевое слово `noexcept` для явного указания этого.

2. **Strong exception safety (строгая гарантия)**: Если операция завершается исключением, состояние программы **не меняется** (как будто операции не было). Пример: `std::vector::push_back` — если при перераспределении памяти не удалось скопировать элемент, вектор остаётся в исходном состоянии.

3. **Basic exception safety (базовая гарантия)**: Если исключение брошено, объект остаётся в **корректном, но неопределённом состоянии** (нет утечек, инварианты соблюдены, но данные могли измениться частично).

Большинство функций стандартной библиотеки предоставляют как минимум базовую гарантию, а многие — строгую.

### 5. Ключевое слово `noexcept`

С C++11 можно указать, что функция **не бросает исключений**:

```cpp
void swap(MyClass& a, MyClass& b) noexcept {
    // реализация, которая гарантированно не бросает исключений
}
```

**Зачем это нужно?**
- **Документирование**: читающий код сразу знает, что функция безопасна.
- **Оптимизации компилятора**: код для `noexcept` функций может быть более эффективным (например, `std::vector` использует перемещение вместо копирования при перераспределении, если перемещающий конструктор `noexcept`).
- **Проверки во время компиляции**: можно узнать, `noexcept` ли выражение, с помощью оператора `noexcept(expr)`.

**Важно:** Если `noexcept` функция всё-таки бросает исключение, вызывается `std::terminate()`, и программа завершается. Поэтому помечайте как `noexcept` только то, что действительно не может бросить исключение (например, деструкторы, move-конструкторы, простые присваивания чисел).

#### Условный `noexcept`

```cpp
template <typename T>
void my_swap(T& a, T& b) noexcept(noexcept(a = std::move(b))) {
    // ...
}
```

Здесь спецификация `noexcept` зависит от того, бросает ли исключения оператор присваивания `T`.

### 6. Исключения в конструкторах и деструкторах

#### Конструкторы

Конструктор может (и часто должен) бросать исключения, если не может создать объект в корректном состоянии. Это **предпочтительный способ** сообщить об ошибке при создании объекта, так как конструктор не может вернуть код ошибки.

```cpp
class File {
    FILE* handle;
public:
    File(const char* filename) : handle(fopen(filename, "r")) {
        if (!handle) {
            throw std::runtime_error("Cannot open file");
        }
    }
    // ...
};
```

Если конструктор бросает исключение, деструктор объекта **не вызывается** (объект не был создан). Однако деструкторы уже сконструированных **членов класса** и **базовых классов** будут вызваны автоматически. Поэтому если вы в конструкторе выделяете ресурсы напрямую (через `new`), нужно быть осторожным: если исключение произойдёт после `new`, но до присвоения умному указателю, произойдёт утечка. **Решение**: всегда использовать умные указатели или RAII-члены.

#### Деструкторы

**Золотое правило: деструкторы НЕ должны бросать исключений.** Почему? Если исключение уже активно (раскрутка стека), и во время вызова деструктора выбрасывается **второе** исключение, программа немедленно вызывает `std::terminate()` и завершается, что почти всегда плохо.

```cpp
class BadClass {
public:
    ~BadClass() {
        throw std::runtime_error("Никогда не делайте так!");   // Категорически запрещено!
    }
};
```

Все стандартные деструкторы неявно помечены как `noexcept(true)`. Если вам нужно выполнить операцию, которая может завершиться ошибкой в деструкторе (например, закрытие файла с проверкой), поглотите исключение внутри деструктора:

```cpp
class GoodClass {
    FILE* f;
public:
    ~GoodClass() {
        try {
            if (fclose(f) != 0) {
                // Записываем в лог, но не бросаем исключение
            }
        } catch (...) {
            // Поглощаем всё
        }
    }
};
```

### 7. Производительность исключений

Существует миф, что исключения "медленные". Это не совсем так. Современные реализации C++ используют модель **нулевых накладных расходов** (zero-cost exception handling). Это означает:

- **При нормальном выполнении** (исключение не брошено) код **не имеет накладных расходов** по сравнению с версией без исключений. Компилятор генерирует дополнительные таблицы (вне кода), которые описывают, как раскручивать стек.
- **Когда исключение брошено**, процесс раскрутки действительно дорогой (поиск обработчика по таблицам), но это исключительная ситуация, которая должна происходить редко.

Таким образом, **исключения следует использовать для действительно исключительных ситуаций** (ошибки ввода-вывода, нехватка памяти, нарушение контракта), а не для управления потоком программы (например, выход из цикла по условию).

### 8. Практические рекомендации

1. **Используйте RAII везде**. Умные указатели, `std::string`, `std::vector`, классы-обёртки для ресурсов — это делает код автоматически безопасным относительно исключений.

2. **Бросайте исключения по значению, ловите по константной ссылке**.
   ```cpp
   throw std::runtime_error("error");   // по значению
   catch (const std::exception& e)      // по ссылке
   ```

3. **Не ловите `...` без крайней необходимости**. Это скрывает все ошибки и затрудняет отладку. Если ловите, делайте это на границах (например, в `main`), чтобы записать в лог и завершить программу аккуратно.

4. **Делайте деструкторы `noexcept` по умолчанию**.

5. **Предпочитайте стандартные классы исключений** или наследуйте от них.

6. **Документируйте исключения, которые может бросать ваша функция** (хотя в C++ нет `throws` как в Java, это хорошая практика в комментариях).

7. **Рассмотрите использование `noexcept` для move-конструкторов и swap**, это повышает эффективность стандартных контейнеров.

8. **Не бойтесь исключений**. В современном C++ с RAII они делают код чище и надёжнее, чем лапша из проверок кодов возврата.

### 9. Пример из реальной разработки: безопасная работа с базой данных

```cpp
#include <iostream>
#include <memory>
#include <stdexcept>
#include <string>

// Фиктивный класс соединения с БД
class DatabaseConnection {
public:
    DatabaseConnection(const std::string& connStr) {
        std::cout << "Connecting to " << connStr << std::endl;
        // Может бросить исключение при неудаче
    }
    
    void execute(const std::string& query) {
        std::cout << "Executing: " << query << std::endl;
        // Может бросить исключение при ошибке SQL
    }
    
    ~DatabaseConnection() {
        std::cout << "Disconnecting" << std::endl;
    }
};

// Транзакция — RAII для автоматического отката при исключении
class Transaction {
    DatabaseConnection& conn;
    bool committed;
public:
    explicit Transaction(DatabaseConnection& c) : conn(c), committed(false) {
        conn.execute("BEGIN TRANSACTION");
    }
    
    void commit() {
        conn.execute("COMMIT");
        committed = true;
    }
    
    ~Transaction() {
        if (!committed) {
            try {
                conn.execute("ROLLBACK");
            } catch (...) {
                // Поглощаем, чтобы не бросить исключение в деструкторе
            }
        }
    }
};

void updateUserData(const std::string& connStr, int userId, const std::string& newName) {
    DatabaseConnection conn(connStr);
    Transaction txn(conn);
    
    // Несколько операций, которые должны быть атомарными
    conn.execute("UPDATE users SET name = '" + newName + "' WHERE id = " + std::to_string(userId));
    conn.execute("INSERT INTO audit_log (user_id, action) VALUES (" + std::to_string(userId) + ", 'name_change')");
    
    txn.commit();
}

int main() {
    try {
        updateUserData("db:example", 42, "Alice");
    } catch (const std::exception& e) {
        std::cerr << "Error updating user: " << e.what() << std::endl;
        // Транзакция автоматически откатится благодаря RAII
        return 1;
    }
    return 0;
}
```

В этом примере:
- `DatabaseConnection` автоматически закрывается при выходе из `updateUserData`.
- `Transaction` автоматически откатывается, если исключение произошло до `commit()`.
- Код в `main` чист и легко читаем, обработка ошибок сосредоточена в одном месте.

### Заключение

Исключения — мощный инструмент обработки ошибок, который в сочетании с RAII позволяет писать надёжный и лаконичный код на C++. Главное — помнить о раскрутке стека, не бросать исключений из деструкторов и использовать умные указатели для автоматического управления ресурсами.
